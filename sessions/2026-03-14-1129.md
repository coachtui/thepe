### project-memory.ts
- `loadProjectMemory()`: fires two parallel queries (project_memory_items + project_source_quality) using service role client. Returns EMPTY_CONTEXT on any error (graceful degradation — tables may not exist yet in older environments). Uses `as any` cast for Supabase client to avoid generated-type mismatch on new tables (consistent with existing pattern in codebase).
- `resolveAliases()`: case-insensitive lookup against aliases. Filters by discipline if provided; if unresolved, falls back to all matches. Sorts by confirmed_by_count desc. Returns only expansions that matched.
- `getSourceQuality()`: specificity scoring: source+discipline+system (score 6) > source+discipline (score 2) > source-only (score 0). Returns cap and modifier from most specific match.
- `sanitizeForPrompt()`: strips non-alphanumeric/punctuation, limits to 200 chars. Applied before alias injection into prompts.
- `formatCalloutPatternsForPrompt()`: formats callout patterns as structured block for Haiku prompt injection.

### query-analyzer.ts
`applyAliasExpansions()` exported as a new pure function:
- Takes QueryAnalysis + expansions map
- Returns new QueryAnalysis with entities.itemName, entities.utilitySystem, entities.componentType replaced (if unambiguous single expansion)
- requestedSystems expanded to all alias values (multi-value OK here)
- Does not re-run classifyQuery() — purely transforms the already-analyzed result

### plan-reader.ts
Added optional `calloutPatterns?: MemoryItem[]` parameter to:
- `runPlanReader()` (public)
- `runPlanReaderWithCandidates()` (internal)
- `inspectPageForQuestion()` (internal)
- `buildInspectionPrompt()` (internal)

Injection happens in the prompt body, after the SHEET line, as a structured block:
```
KNOWN PROJECT ABBREVIATIONS (use these to interpret callout labels on this drawing):
  - "HORIZ DEFL" = horizontal deflection fitting
  - "MJ BEND" = mechanical joint bend
```
Only injected when calloutPatterns is non-empty.

### chat-handler.ts
Step 0 (before analyzeQuery): loadProjectMemory(projectId) — parallel DB query, non-blocking if tables empty.
Step 0.5 (after analyzeQuery): resolveAliases on 3 entity fields + requestedSystems, then applyAliasExpansions if any expansions found.
calloutPatterns passed as 5th arg to runPlanReader.

## Decisions Made
- `as any` cast for Supabase client in project-memory.ts: generated types don't include new tables until migration is applied and types are regenerated. Consistent with pattern used throughout codebase.
- `loadProjectMemory()` uses service role client (not passed-in supabase): memory loads need service role access since project_memory_items has no anon read policy. This is consistent with CLAUDE.md rule "service role client for all writes and backend queries."
- Alias expansion is applied after, not inside, analyzeQuery() — keeps analyzeQuery() synchronous and pure.
- Plan reader `calloutPatterns` is optional — callers without memory context pass nothing; behavior is identical to pre-Phase-7B.
