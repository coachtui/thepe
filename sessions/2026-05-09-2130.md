# Session 2026-05-09-2130

## Goal
Implement PDF visual line reconstruction for ingestion harness. Fix root cause of pdfjs producing one 1000–3000 char blob per page, which prevents line-level SD code extraction and nearby-SD association.

## Files Created or Modified

### New: `src/lib/parsers/pdf-line-reconstruction.ts`
- `groupPdfTextItemsIntoLines(textItems, options?)` — pure function. Groups raw PDF.js text items by Y coordinate (PDF baseline). Items within `yTolerance` (default 3 units) are grouped into the same visual line, then sorted left-to-right by X. Space inserted when inter-item gap > 1 unit. Returns `{ lines, rawItemCount }`.
- `parseDocumentWithLineReconstruction(filePath, fileName, options?)` — full parser. Same page-break format as `parseDocumentWithPdfJs` but uses `groupPdfTextItemsIntoLines` per page. Returns `lineMetrics` with before/after comparison values.
- Mirrors pdfjs-dist worker setup from `pdfjs-parser.ts` exactly.

## SD Coverage Delta (Key Finding)

**Before line reconstruction:** 26.6% SD coverage
**After line reconstruction:** 15.7% SD coverage

This is a **regression**, not caused by a bug in line reconstruction. Root cause:

Before (blob behavior): `items.join(' ')` on each page accidentally concatenated SD code text ("SD-03 Product Data") into the same text blob as nearby item descriptions. `extractSdCode()` found them inline → 26.6% was inflated by accidental concatenation.

After (correct behavior): SD code lines are now visually separate. Nearby-SD association picks up 212 associations (up from 0), but `MAX_DISTANCE = 2` in `nearby-sd-association.ts` is too narrow for the UFGS table format, where SD codes can appear in a table column 5–10 lines from the description row.

**Evidence:** 477 SD-only lines detected, but only 212 associated. 265 SD-only lines have no nearby item within 2 lines — they're in table column positions or summary sections.

## Decisions

- Harness-only integration confirmed safe. `pdfjs-parser.ts` (production path) is untouched.
- Did not increase `MAX_DISTANCE` — that optimization is follow-on work; don't overfit.
- Did not build DD-form appendix parser — per user spec.

## Unresolved Items

- **MAX_DISTANCE = 2 too narrow for UFGS format.** The 477 SD-only lines → 212 associations gap is the next SD coverage lever. Increasing to 4–5 should capture more. But needs real-format analysis first.
- **UFGS SD code placement pattern:** The spec puts SD codes in a dedicated "SUBMITTALS" paragraph column format, not inline. Understanding the exact column structure could unlock a table-aware extraction path.
- **26.6% was inflated.** The true pre-reconstruction "honest" coverage might be even lower than 15.7% in some sense — 26.6% was an artifact.
