## Decisions and Analysis

**Why did SD coverage go down despite distance=5?**

The skip-if-multiple-candidates rule is more conservative than the old first-wins approach. In the UFGS spec, 138 SD-only lines have multiple adjacent submittal items in their window. Under first-wins, those associations would fire for the first item found. Under skip-if-multiple, they're all skipped.

Backward associations collapsed from 111 → 21 for the same reason: the backward scan now encounters multiple candidates within the window and skips.

**Root cause of 138 multi-cand skips in UFGS:** The UFGS "SUBMITTALS" section structure frequently places one SD category header above a block of 3–5 items that all belong to that category. Example:
```
SD-03 Product Data        ← SD-only line
Concrete mix design       ← item 1
Aggregate gradation data  ← item 2
Admixture data            ← item 3
```
The skip-if-multiple prevents association here, but in reality ALL three items genuinely share SD-03.

**Is this a problem?** The rule is intentionally conservative for safety (avoids the wrong item getting an SD code). For UFGS PDFs, the correct fix is a table-aware multi-item association: when an SD-only line precedes a block of items, apply the SD code to ALL of them. This is a different algorithm from the current forward/backward scan.

## Unresolved Items

- **UFGS multi-item block association:** 138 multi-cand skips = a block of ≥2 items under one SD category header. The correct fix is to apply the SD code to ALL items in the block, not skip. Requires a new pass that detects "item blocks" (consecutive submittal lines without a boundary or another SD-only line) and applies the SD code uniformly.
- **SD coverage plateau at ~15%:** Neither distance increase nor skip-if-multiple significantly improved UFGS coverage. The fundamental issue is the UFGS table structure, not the search distance.
- **Adversarial fixture regression (68.2%):** Intentional; 2 multi-cand skips are correct behavior.
