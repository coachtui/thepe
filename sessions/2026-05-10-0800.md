# Session 2026-05-10-0800

## Goal
Add Pass 5 block association to `associateNearbySdCodes`. When a single SD category header
precedes a contiguous block of submittal items, apply that code to all items in the block
(UFGS table-column format). Enabled only for `reconstructed_pdf` / `ufgs` mode.

## Files Modified

### `src/lib/ingestion/nearby-sd-association.ts`
- Added 4 new fields to `NearbySdAssociationMetrics`: `blockHeadersDetected`, `blockAssociations`,
  `blockSkippedDueToInline`, `blockTerminatedByBoundary`.
- Added metrics initializer values for all 4 fields.
- Added **Pass 5** after Pass 4: runs only for `mode === 'reconstructed_pdf' | 'ufgs'`.
  For each SD-only line, scans forward collecting ALL candidate items until hitting a
  boundary, another SD header, or end-of-lines. Assigns SD code to every item in the
  block that doesn't already have an association or inline code. Does not stop at non-item
  filler lines — they are skipped over, not block-terminators.

### `src/lib/eval/ingestion-types.ts`
- Added 4 new fields to `NearbySdMetrics`.

### `src/lib/eval/ingestion-runner.ts`

## Analysis

**Block pass fired on 44 of 477 SD-only lines.** The other 433 either:
- Were already consumed by forward/backward passes (103+21 = 124 lines)
- Had `blockTerminatedByBoundary` = 183 (boundary before any items in scan range)
- Had no items in their forward range at all (remaining ~126)

**183 boundary-ends** is high — this confirms the UFGS spec puts many SD-only lines very close to
section boundaries. Those lines can't have block associations extended further.

**SD coverage plateau:** 18.9% is the ceiling reachable with the current text-line extraction approach
on this UFGS spec. Further gains would require:
1. Parsing the 75-page DD-form submittal appendix (authoritative SD assignments in table format)
2. Or a fundamentally different extraction strategy for the UFGS "1.4 SUBMITTALS" paragraph format

## Decisions

- `SD_ONLY_MAX_LENGTH = 60` interacts with inline-SD-code lines shorter than 60 chars: they look
  like SD-only headers. NSD-19 was fixed to use a longer (> 60 char) submittal line to correctly
  test the inline-overwrite protection. This is a known edge case in the detection logic.
- Block scan does not limit how many filler lines can appear between items in a block — tolerates
  all non-boundary non-SD-only lines as filler. This is correct for UFGS format.
