# Session 2026-05-06-0748

## Goal

Refactor spec document extraction to use section-batched processing and incremental persistence. Avoid the single 25+ minute Inngest step that causes stale Supabase connections on large spec documents (4,987 chunks, ~80 sections).

---

## Files Modified

### `src/lib/chat/spec-extraction-pipeline.ts`
Added three exported types and two exported functions:
- `SectionManifestEntry` — lightweight, JSON-serialisable per-section record (sectionNumber, sectionTitle, bodyText, sourceChunkIds, sourcePageNumbers). Safe to store as Inngest step result.
- `DiscoverSpecSectionsResult` — return type for `discoverSpecSections`.
- `ExtractSectionBatchResult` — return type for `extractSectionBatch`.
- `discoverSpecSections(input)` — runs all non-LLM pipeline phases (concat, CSI regex, dedup, cap) and returns the section manifest. No LLM calls. Designed to run in one fast Inngest step.
- `extractSectionBatch(input)` — takes a batch of `SectionManifestEntry[]`, constructs synthetic `SectionSlice` objects, and runs `extractSection` (with LLM) for each. Returns `SpecSectionExtractionResult[]` plus batch metrics.

The existing `runSpecExtractionPipeline` is unchanged (harness still uses it).

### `src/lib/chat/spec-extraction-persistence.ts`
Added `skipDelete?: boolean` to `PersistSpecExtractionOptions`. When `true`, the full document-level delete step is skipped. Used in batch mode where the caller manages deletion (either one-time full delete at start, or per-batch scoped delete for retry idempotency).

### `src/inngest/functions/spec-extract-document.ts`
Full rewrite from 3-step monolith to batched multi-step function:

| Step | Name | What it does |
|---|---|---|
| 1 | `load-document` | Verify doc type + status (same as before) |
| 2 | `discover-sections` | Load all chunks (paginated), run `discoverSpecSections`, return manifest |
| 3 | `delete-existing` | One-time full delete of spec entities for this document |
| 4..N | `extract-batch-{i}` | LLM extraction for 5 sections, scoped delete, immediate persist |

Batch size: `BATCH_SIZE = 5` sections per step.

---

## New Batch Execution Model

```
Inngest: spec/document.extract
  → step 1: load-document (fresh client, fast DB read)
  → step 2: discover-sections (fresh client, paginated chunk load + regex; no LLM)
  → step 3: delete-existing (fresh client, one-time full delete)
  → step 4: extract-batch-0 (fresh client, LLM × 5 sections, scoped delete, persist)
  → step 5: extract-batch-1 (fresh client, LLM × 5 sections, scoped delete, persist)
  → ...
  → step N: extract-batch-{k} (last batch, ≤5 sections)
  → return aggregated summary
```

For ~80 sections: ~16 batches → 16 independent steps, each ~30-75s of LLM calls with a fresh DB connection.

---

## Idempotency

| Scenario | Behavior |
|---|---|
| First run | `delete-existing` clears old state; each batch inserts fresh |
| Inngest retry of a failed batch | `delete-existing` is memoised (skipped); batch scoped-delete cleans partial inserts before re-insert |
| User re-triggers extraction | New function invocation: `delete-existing` runs fresh, replaces all previous spec state |
| New run with different section count | `delete-existing` wipes all old sections; only newly-extracted sections are re-inserted |

Per-batch scoped delete uses LIKE prefix: `SPEC_{sectionNumber}%` — covers both the section entity (`SPEC_01_33_00`) and all its requirement entities (`SPEC_01_33_00_REQ_...`).

---

## Persistence Behavior

- `skipDelete: true` in persistence helper → no full document delete per batch
- Each batch owns its own scoped delete before calling persistence
- Citations and findings still cascade-delete when their requirement entity is deleted (via ON DELETE CASCADE schema)
- Final summary aggregates across all batch outcomes: sectionsAttempted, sectionsSucceeded, requirementsWritten, citationsWritten, findingsWritten, totalCostUsd, failedBatches

---

## Retry / Failure Behavior

- Any batch step that throws causes Inngest to retry that step (not the whole function)
- Completed batch steps are memoised — their DB inserts remain; they are not re-run
- Per-batch scoped delete handles the case where a step failed mid-insert and left partial rows
- If all retries exhaust, `onFailure` logs `[FAILED-FINAL]`
- A fresh user-triggered run is always safe: `delete-existing` runs at the top

## Unresolved Items

- `BATCH_SIZE = 5` was chosen conservatively. Can be tuned if LLM rate limits allow.
- On Vercel production, each batch step runs in a fresh serverless invocation with a fresh DB connection — the local 25-minute timeout root cause is fully eliminated.
- The `discover-sections` step stores `bodyText` in the section manifest (step result). For very large specs (~80 sections × 10K chars average), the step result is ~800KB — within Inngest limits. If this ever causes issues with extremely large specs, the manifest can be moved to Supabase and referenced by ID.
- Manual endpoint behavior is unchanged (`extract-specs` POST still fires `spec/document.extract`).
- No auto-trigger added.
