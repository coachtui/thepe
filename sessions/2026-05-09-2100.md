## Key Discoveries

1. **pdfjs uses `---PAGE-BREAK---` not `\f`**: The ingestion harness initially saw "1 page" because the normalization was splitting on `\f`. Fixed by detecting the separator dynamically.

2. **Majority prefix (95%) over exact prefix**: Two outlier lines used "Project" (mixed case) and omitted the facility number "1644749". Using 100% matching would never find the full header. The 95% threshold catches the full "FY22 MILCON PROJECT PN 080133 AMMUNITION STORAGE 1644749 WEST LOCH, HAWAII" prefix correctly.

3. **SD coverage gap is not contamination**: After correct prefix stripping, SD% remains 26.6%. Root cause: UFGS specs list SD codes on separate lines from item descriptions; the extractor matches SD codes inline with the item text. The 75-page DD-form submittal appendix (table format) has the authoritative SD code list but is not parsed by the current extractor.

4. **285 warnings are still real**: Most are `conditional_approval_authority` findings. These existed before contamination — they reflect real ambiguity in the spec language, not header artifacts.

## Unresolved Items
- UFGS SD code extraction gap: SD codes appear on dedicated lines in the spec body, not inline with item descriptions. Consider multi-line context extraction.
- 75-page DD-form submittal appendix: table-format, authoritative, not parsed. Highest-value extraction target for this spec type.
- Frequency-based header detection never triggered (pdfjs produces 1 text block with ---PAGE-BREAK--- strings; the page-split works but header patterns need to appear in the top 3 lines per page to be detected, which they do after the page separator fix — but the DoD spec was already handled by prefix detection).
